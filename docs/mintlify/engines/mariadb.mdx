## Overview

| Property | Value |
|----------|-------|
| Engine ID | `mariadb` |
| Wire Protocol | MySQL wire protocol |
| Default Port | 3306 |
| Status | Supported |
| Classifier | AST-based (`vitess/sqlparser`) |
| Shadow | Docker container (MariaDB image) |

MariaDB speaks the MySQL wire protocol, so Mori reuses the MySQL engine implementation. The MariaDB engine overrides only the `Init` method to use a MariaDB Docker image for the shadow container -- all proxy logic, classification, and merging is shared with MySQL.

## Connection Parameters

| Field | Description | Default |
|-------|-------------|---------|
| `host` | Database host | -- |
| `port` | Database port | `3306` |
| `user` | Username | -- |
| `password` | Password | -- |
| `database` | Database name | -- |
| `ssl_mode` | SSL mode | -- |

## Implementation Details

### Why a Separate Engine?

MariaDB and MySQL have diverged enough that using a MySQL shadow container for MariaDB production can cause issues. Specifically, MySQL 8.0's `mysqldump` queries `COLUMN_STATISTICS` which MariaDB does not support. Mori uses the `mariadb:` Docker image to avoid these compatibility problems.

### Query Classification

Identical to MySQL -- uses `vitess/sqlparser`. MariaDB's SQL dialect is MySQL-compatible for the statements Mori classifies.

### Shadow Strategy

Docker container running MariaDB, version-matched to production via `SELECT VERSION()`.

- Image: `mariadb:<version>` (matched to prod)
- Schema dump and replay identical to MySQL

## Example Usage

```bash
# Initialize with MariaDB
mori init --from "mysql://user:pass@prod-host:3306/mydb"

# Start the proxy
mori start

# Connect with the MySQL client (MariaDB is wire-compatible)
mysql -h 127.0.0.1 -P 3306 -u user -p mydb
```

## Known Limitations

<Note>
MariaDB has **zero known limitations**. All E2E tests pass without skips. MariaDB avoids the `only_full_group_by` issue that affects MySQL because MariaDB does not enforce that mode by default.
</Note>
