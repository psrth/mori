## Overview

| Property | Value |
|----------|-------|
| Engine ID | `cockroachdb` |
| Wire Protocol | pgwire (PostgreSQL v3) |
| Default Port | 26257 |
| Status | **Stable** |
| Classifier | AST-based (`pg_query_go`) |
| Shadow | Docker container (CockroachDB) |

CockroachDB speaks the pgwire protocol, so Mori reuses the entire PostgreSQL engine implementation. The CockroachDB engine is a thin adapter that overrides only the engine ID -- all proxy logic, classification, and read/write merging is shared with PostgreSQL.

## Connection Parameters

| Field | Description | Default |
|-------|-------------|---------|
| `host` | Database host | -- |
| `port` | Database port | `26257` |
| `user` | Username | -- |
| `password` | Password | -- |
| `database` | Database name | -- |
| `ssl_mode` | SSL mode (`disable`, `require`, `verify-ca`, `verify-full`) | `disable` |

## Implementation Details

### Query Classification

Identical to PostgreSQL -- uses `pg_query_go` for full AST-based classification. CockroachDB's SQL dialect is PostgreSQL-compatible, so the same parser handles it natively.

### Shadow Strategy

Docker container running CockroachDB, version-matched to production. Schema is dumped and replayed on shadow during `mori init`.

### Shared Implementation

CockroachDB's engine is defined as an embedded struct of the PostgreSQL engine:

```go
type crdbEngine struct {
    pgEngine // Reuses all PostgreSQL logic
}
```

This means CockroachDB gets every feature PostgreSQL has -- extended query protocol, transaction support, DDL tracking, write guard -- without any separate code paths.

## Example Usage

```bash
# Initialize with CockroachDB
mori init --from "postgres://user:pass@crdb-host:26257/mydb?sslmode=require"

# Start the proxy
mori start

# Connect with any PostgreSQL client
psql "postgres://user:pass@127.0.0.1:26257/mydb"
```
