## Overview

| Property | Value |
|----------|-------|
| Engine ID | `duckdb` |
| Wire Protocol | pgwire (exposed by Mori) |
| Default Port | N/A (embedded) |
| Status | Supported |
| Classifier | Regex-based |
| Shadow | Local file copy |

DuckDB is an embedded analytical database. Like SQLite, Mori exposes it over the pgwire protocol so applications connect with a PostgreSQL driver. The shadow is a local file copy -- no Docker container needed.

## Connection Parameters

| Field | Description |
|-------|------------|
| `file_path` | Path to the DuckDB database file (e.g., `/path/to/db.duckdb`) |

Also supports URI format: `duckdb:///path/to/db.duckdb` and in-memory: `:memory:`

## Implementation Details

### Query Classification

Uses regex-based classification. DuckDB SQL is approximately 90% PostgreSQL-compatible, so the classifier handles PG-style syntax with DuckDB-specific extensions:

- Standard DML (SELECT, INSERT, UPDATE, DELETE)
- DuckDB-specific functions and types
- DDL (CREATE, ALTER, DROP)
- Transaction control (BEGIN, COMMIT, ROLLBACK)

### Shadow Strategy

Local file copy, identical to SQLite. The shadow is stored at `.mori/shadow.db` in the project's mori directory.

During `mori init`:
1. The prod DuckDB file is copied to the shadow location
2. Table metadata is extracted and persisted

<Note>
No Docker required. Both SQLite and DuckDB use the embedded file-copy strategy for the shadow.
</Note>

### pgwire Exposure

DuckDB has no native network protocol. Mori opens both prod and shadow DuckDB files directly and exposes them through a pgwire listener. Connect with any PostgreSQL client.

## Example Usage

```bash
# Initialize with a DuckDB file
mori init --from "/path/to/analytics.duckdb"

# Start the proxy
mori start

# Connect with psql or any PostgreSQL driver
psql "postgres://localhost:5432/main"
```

## Known Limitations

<Warning>
**Transaction COMMIT discards shadow writes** -- Same as SQLite. `BEGIN; INSERT ...; COMMIT` fails. Use auto-commit writes instead.
</Warning>

**JOIN_PATCH doesn't reflect shadow UPDATEs** -- Multi-table JOINs where one table has shadow-modified rows return stale values. Single-table reads work correctly.

**`LIST()` aggregate returns unscannable array type** -- DuckDB's `LIST()` function returns a native array type that the proxy's scanner can't deserialize. Other aggregate functions (SUM, AVG, COUNT, etc.) work fine.
