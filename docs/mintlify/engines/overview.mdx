Mori supports 9 database engines across SQL, NoSQL, and embedded categories. Each engine uses the real wire protocol -- your application connects to Mori with its standard driver, no code changes required.

## Engine Matrix

| [PostgreSQL](/engines/postgres) | SQL | pgwire | 5432 | Docker container | **Stable** |

| [Firestore](/engines/firestore) | NoSQL | gRPC | N/A | Emulator container | Supported |

## By Category

### SQL Engines (pgwire)

PostgreSQL and CockroachDB speak the pgwire protocol. Mori uses `pg_query_go` (a real PostgreSQL parser) for AST-based query classification. These engines have the most mature proxy implementation with zero known limitations.

### SQL Engines (MySQL wire)

MySQL and MariaDB use the MySQL wire protocol. Mori uses `vitess/sqlparser` for AST-based classification. MariaDB reuses the MySQL engine implementation entirely -- the only difference is that MariaDB uses a `mariadb:` Docker image for the shadow container.

### SQL Engines (TDS)

MS SQL Server speaks the TDS (Tabular Data Stream) protocol. Mori uses regex-based classification for T-SQL since no Go-native TDS parser exists. Supports `sp_executesql` RPC calls alongside standard SQL batches.

### Embedded Engines

SQLite and DuckDB are embedded databases (no server process). Mori exposes them over the pgwire protocol so applications can connect with any PostgreSQL driver. The shadow is a local file copy rather than a Docker container.

### NoSQL Engines

Redis uses the RESP protocol with a command-based classifier. Firestore uses gRPC with a method-name classifier. Both support full copy-on-write semantics with delta tracking.

<Info>
All 9 engines share the same copy-on-write model: reads from prod, writes to shadow, transparent merging. The wire protocol and classification strategy differ per engine, but the invariants are identical.
</Info>
