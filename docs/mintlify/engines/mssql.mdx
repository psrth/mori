## Overview

| Property | Value |
|----------|-------|
| Engine ID | `mssql` |
| Wire Protocol | TDS (Tabular Data Stream) |
| Default Port | 1433 |
| Status | Supported |
| Classifier | Regex-based |
| Shadow | Docker container (`mcr.microsoft.com/mssql/server`) |

MS SQL Server uses Microsoft's TDS protocol. Mori implements a TDS proxy that handles login negotiation, SQL batch messages, and RPC requests (including `sp_executesql`).

## Connection Parameters

| Field | Description | Default |
|-------|-------------|---------|
| `host` | Database host | -- |
| `port` | Database port | `1433` |
| `user` | Username | -- |
| `password` | Password | -- |
| `database` | Database name | -- |
| `encrypt` | Encryption setting | -- |
| `trust_server_cert` | Trust server certificate | -- |

## Implementation Details

### Query Classification

Uses regex-based classification for T-SQL. No Go-native TDS/T-SQL AST parser exists, so Mori uses pattern matching to identify:

- Standard DML (SELECT, INSERT, UPDATE, DELETE)
- T-SQL specific syntax (TOP, OUTPUT, MERGE)
- `sp_executesql` RPC calls (parameterized queries)
- DDL (CREATE, ALTER, DROP)
- Transaction control (BEGIN TRAN, COMMIT, ROLLBACK)
- `EXEC`/`EXECUTE` statements

### Shadow Strategy

Docker container running the official Microsoft SQL Server Linux image.

- Image: `mcr.microsoft.com/mssql/server:2022-latest`
- SA password is auto-generated during init
- Schema is dumped and replayed on the shadow container

### TDS Protocol Support

The proxy handles:

- TDS login7 packets for authentication
- SQL batch messages (`TDS_SQL_BATCH`)
- RPC requests (`TDS_RPC_REQUEST`) including `sp_executesql`
- Attention signals for query cancellation
- SSL/TLS negotiation (ENCRYPT_ON, ENCRYPT_NOT_SUP)

## Example Usage

```bash
# Initialize with MSSQL
mori init --from "sqlserver://sa:password@prod-host:1433?database=mydb&encrypt=true&trustservercertificate=true"

# Start the proxy
mori start

# Connect with sqlcmd
sqlcmd -S 127.0.0.1,1433 -U sa -P password -d mydb
```

## Known Limitations

<Warning>
MSSQL has the most known limitations of any engine (6 skipped E2E tests).
</Warning>

**Transaction COMMIT discards shadow writes** -- `BEGIN TRAN; INSERT ...; COMMIT` fails because the TDS proxy doesn't properly persist shadow deltas inside explicit transaction blocks. Auto-commit writes work fine.

**`sp_executesql` SELECT with parameters not resolved** -- Parameterized queries via `sp_executesql` (used by default in .NET drivers) return incorrect merged-read results because `@p1`, `@p2` placeholders can't be resolved. Direct SQL strings work correctly.

**`sp_executesql` UPDATE parameters not hydrated** -- Same root cause as above but for write operations. The proxy can't extract parameter values from TDS parameter tokens for hydration.

**JOIN_PATCH doesn't reflect shadow UPDATEs** -- Multi-table JOINs where one table has shadow-modified rows return stale (prod) values. Single-table reads work correctly.

**JOIN_PATCH doesn't include shadow INSERTs** -- Rows inserted into shadow don't appear in JOIN results.

**`sp_rename` fails on shadow tables** -- `EXEC sp_rename 'table.col', 'new_col', 'COLUMN'` fails due to ambiguity between prod and shadow table names.

<Tip>
For .NET applications: if you hit `sp_executesql` issues, consider using `CommandType.Text` with inline parameters instead of `CommandType.StoredProcedure` for queries routed through Mori.
</Tip>
