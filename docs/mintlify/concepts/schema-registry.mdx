## What is the Schema Registry?

The Schema Registry tracks per-table structural differences between the Shadow and Prod schemas. When you run DDL on Shadow (the only place DDL executes), the schemas diverge. The registry records exactly what changed so Mori can adapt Prod rows during reads and hydration.

## Why It's Needed

DDL runs on Shadow only — production schema is never modified. But reads still come from Prod, which has the original schema. The Schema Registry bridges this gap by transforming Prod rows to match the Shadow schema before merging.

## Operations Tracked

### ADD COLUMN

- **Registry records**: column name, type, default value
- **Read adaptation**: Prod rows get NULL or the default value injected for the new column
- **Hydration adaptation**: hydrated rows get NULL or the default value

### DROP COLUMN

- **Registry records**: dropped column name
- **Read adaptation**: dropped column is stripped from Prod results
- **Hydration adaptation**: dropped column is excluded when copying rows

### RENAME COLUMN

- **Registry records**: `old_name -> new_name` mapping
- **Read adaptation**: column is renamed in Prod results
- **Hydration adaptation**: column is renamed when inserting into Shadow

### ALTER TYPE

- **Registry records**: `old_type -> new_type` mapping
- **Read adaptation**: value is cast to the new type in Prod results
- **Hydration adaptation**: value is cast when inserting into Shadow

### CREATE TABLE

- **Registry records**: shadow-only table
- **Read/hydration**: N/A — table only exists in Shadow

### DROP TABLE

- **Registry records**: table marked as dropped
- **Read adaptation**: queries return "table not found"

### CREATE INDEX

- **Registry records**: shadow only, silent
- **Read/hydration**: N/A — indexes are internal to Shadow

## How Adaptation Works

During a merged read, the Merge Engine consults the Schema Registry before combining results:

1. Fetch Prod rows (original schema)
2. For each registered divergence, transform the row: inject NULLs, strip columns, rename, cast types
3. Adapted Prod rows now match Shadow schema
4. Merge with Shadow rows and return unified result
