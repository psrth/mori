# Mori

**Database Virtualization Layer**
_Copy-on-write semantics for production databases._

---

## 1. Overview

Mori is a CLI tool that sits between a local application and a production database. It proxies all database traffic, serving reads from production and capturing writes locally. The application connects to Mori as if it were a normal database — no code changes, no config changes beyond a connection string swap.

All mutations are local. Production is never written to. Break something, reset, start over.

### 1.1 Core Invariants

- **Production is read-only.** Mori never issues a write to the upstream database. Ever.
- **Shadow is the mutation layer.** All writes go to a local database instance (the Shadow) that mirrors production's schema.
- **The app sees one unified database.** Reads merge Shadow and Production transparently. The app cannot distinguish Mori from a real database.
- **Reset is instant.** Wiping the Shadow and its metadata restores a clean-slate view of production.

### 1.2 Terminology

| Term                | Definition                                                                                                  |
| ------------------- | ----------------------------------------------------------------------------------------------------------- |
| **Prod**            | The upstream production database. Read-only from Mori's perspective.                                        |
| **Shadow**          | A local database instance (same engine as Prod) holding all local mutations. Starts schema-only, zero rows. |
| **Proxy**           | The Mori process that intercepts wire protocol traffic between the App and Prod/Shadow.                     |
| **Taint Map**       | A set of `(table, primary_key)` pairs identifying rows that have been modified locally.                     |
| **Tombstone Set**   | A set of `(table, primary_key)` pairs identifying rows that have been deleted locally.                      |
| **Schema Registry** | A record of structural differences between Shadow and Prod schemas, introduced by local DDL.                |
| **Hydration**       | The act of copying a row from Prod into Shadow so it can be mutated locally.                                |

## 2. Lifecycle

### 2.1 Initialization

```
mori init --from <prod_connection_string>
```

1. Connect to Prod. Run `pg_dump --schema-only` (or equivalent for the target engine).
2. Spin up a local Shadow container (same engine, same version as Prod).
3. Apply the schema dump to Shadow.
4. Offset all auto-increment sequences in Shadow:
   - `Shadow Sequence Start = Prod Current Max + 1,000,000`
   - This guarantees locally-created PKs never collide with Prod PKs.
5. Initialize empty Taint Map, Tombstone Set, and Schema Registry.
6. Write a `.mori/` config directory with connection metadata.

### 2.2 Start

```
mori start [--port <port>]
```

1. Start the Shadow container (if not running).
2. Start the Proxy, listening on `localhost:<port>` (default: the engine's standard port).
3. Establish a read-only connection pool to Prod.
4. Establish a read-write connection pool to Shadow.
5. Load Taint Map, Tombstone Set, and Schema Registry into memory.
6. Begin accepting client connections.

### 2.3 Reset

```
mori reset
```

1. Drop all data in Shadow (truncate all tables).
2. Re-sync schema from Prod (`pg_dump --schema-only`, re-apply).
3. Re-offset sequences.
4. Clear Taint Map, Tombstone Set, and Schema Registry.

### 2.4 Stop

```
mori stop
```

1. Drain active connections.
2. Persist Taint Map, Tombstone Set, and Schema Registry to disk.
3. Stop the Proxy.
4. Optionally stop the Shadow container (`--keep-shadow` to leave it running).

## 3. Proxy Architecture

### 3.1 Wire Protocol Layer

The Proxy implements the target database's wire protocol (e.g., Postgres v3 protocol). The application connects to Mori's listening port using a standard database driver. From the app's perspective, Mori is the database.

The Proxy handles:

- Connection handshake and authentication (pass-through or local auth).
- Statement parsing (simple query, extended query / prepared statements).
- Result serialization back to the client.

### 3.2 Operation Classifier

Every incoming statement is parsed minimally to determine:

1. **Operation type**: `READ`, `WRITE`, `DDL`, `TRANSACTION_CONTROL`, or `OTHER`.
2. **Affected tables**: The set of table names referenced in the statement.
3. **Affected keys** (when extractable): Primary key values from WHERE clauses for point lookups.

The parser does not need to understand full query semantics. It extracts operation type and table names. This can be done via lightweight SQL parsing (AST-level, not planning-level).

### 3.3 Router

The Router uses the classifier output + Taint Map + Tombstone Set to decide the execution strategy:

| Operation               | Any Table Tainted? | Strategy                                          |
| ----------------------- | ------------------ | ------------------------------------------------- |
| `SELECT`                | No                 | **Prod Direct** — forward to Prod, return result. |
| `SELECT`                | Yes                | **Merged Read** — see §4.                         |
| `INSERT`                | —                  | **Shadow Write** — execute on Shadow only.        |
| `UPDATE`                | —                  | **Hydrate + Shadow Write** — see §5.              |
| `DELETE`                | —                  | **Shadow Delete** — see §6.                       |
| `DDL`                   | —                  | **Shadow DDL** — see §7.                          |
| `BEGIN/COMMIT/ROLLBACK` | —                  | **Transaction Manager** — see §8.                 |

## 4. Read Path (Merged Reads)

Merged reads are the core complexity of Mori. They occur when a `SELECT` touches at least one table that has entries in the Taint Map or Tombstone Set.

### 4.1 Single-Table Read

**Input:** `SELECT * FROM users WHERE active = true ORDER BY created_at LIMIT 20`
**Condition:** `users` has tainted rows.

**Steps:**

1. **Shadow Query.** Execute the original query against Shadow verbatim. Shadow contains only locally-created or hydrated rows, so this returns the local subset.

2. **Sanitize Query for Prod.** Remove any WHERE predicates that reference columns existing only in Shadow (tracked via Schema Registry). Preserve everything else.

3. **Prod Query.** Execute the sanitized query against Prod.

4. **Filter Prod Results.** For each row returned from Prod:
   - If `(table, pk)` exists in Taint Map → discard (Shadow has the authoritative version).
   - If `(table, pk)` exists in Tombstone Set → discard (row is locally deleted).

5. **Adapt Prod Results.** If Schema Registry indicates divergence for this table:
   - Inject `NULL` (or column default) for columns added in Shadow but absent in Prod.
   - Drop columns that were removed in Shadow but present in Prod.
   - Apply renames/type casts as registered.

6. **Merge.** Combine Shadow results + filtered/adapted Prod results.

7. **Re-apply Ordering and Limits.** Sort the merged set by the original `ORDER BY`. Apply `LIMIT`/`OFFSET` to the merged set.

### 4.2 Over-Fetching Strategy

Because Prod results are post-filtered (tainted and tombstoned rows removed), a `LIMIT N` on the Prod query may yield fewer than N usable rows after filtering.

**Strategy:** Over-fetch from Prod by a factor of `LIMIT + |tombstones_for_table| + |taints_for_table|`. If after filtering the result set is still short, issue a follow-up query with an offset. Cap at 3 iterations to bound latency.

For queries with no LIMIT, this is not an issue — all rows are fetched and filtered.

## 5. Write Path — INSERT

**Input:** `INSERT INTO users (name, email) VALUES ('alice', 'alice@example.com')`

1. Execute the INSERT on Shadow only.
2. Shadow's offset sequence assigns a PK in the local range (e.g., `1,000,001`).
3. Return the result (including generated PK) to the app.

No Taint Map entry is needed for pure inserts — the row only exists in Shadow. However, if the app later queries by this PK, the Router must know to check Shadow. Since the PK is in the offset range (`> 1,000,000`), the Router can fast-path: **any PK in the local range is always routed to Shadow.**

---

## 6. Write Path — UPDATE

**Input:** `UPDATE users SET email = 'new@example.com' WHERE id = 42`

1. **Check Shadow.** Does `(users, 42)` exist in Shadow?
   - **Yes:** Row was previously hydrated or locally created. Apply UPDATE to Shadow.
   - **No:** Row exists only in Prod. **Hydrate first:**
     a. Fetch the row from Prod: `SELECT * FROM users WHERE id = 42`.
     b. Adapt the row (inject NULLs for new columns, drop removed columns per Schema Registry).
     c. Insert the adapted row into Shadow.
2. Apply the UPDATE to the row in Shadow.
3. Add `(users, 42)` to Taint Map.
4. Return result to app.

### 6.1 Bulk Updates

`UPDATE users SET active = false WHERE last_login < '2020-01-01'`

1. Determine affected rows. Run `SELECT id FROM users WHERE last_login < '2020-01-01'` against **both** Prod and Shadow.
2. For each affected Prod row not already in Shadow: hydrate.
3. Apply the UPDATE on Shadow.
4. Add all affected PKs to Taint Map.

This is expensive for large result sets. Acceptable for a development tool. Prod is never mutated regardless.

---

## 7. Write Path — DELETE

**Input:** `DELETE FROM users WHERE id = 42`

1. If `(users, 42)` exists in Shadow → delete it from Shadow.
2. Add `(users, 42)` to Tombstone Set. (Even if the row was Shadow-only, tombstoning is harmless and keeps logic uniform.)
3. Return result to app.

For locally-created rows (PK in offset range), the tombstone is technically unnecessary (the row doesn't exist in Prod to filter out), but adding it keeps the delete path branchless.

---

## 8. DDL Path — Schema Changes

**Input:** `ALTER TABLE users ADD COLUMN phone TEXT`

1. Execute the DDL on Shadow only.
2. Update Schema Registry:
   ```
   users:
     added: [{name: phone, type: TEXT, default: NULL}]
   ```
3. Return success to app.

### 8.1 Supported Schema Operations

| DDL             | Registry Action                       | Inbound Adapter (Hydration)                        | Outbound Adapter (Reads)                                 |
| --------------- | ------------------------------------- | -------------------------------------------------- | -------------------------------------------------------- |
| `ADD COLUMN`    | Record column name, type, default.    | Hydrated rows get `DEFAULT`/`NULL` for new column. | Prod rows get `NULL`/default injected.                   |
| `DROP COLUMN`   | Record dropped column name.           | Hydrated rows: dropped column excluded.            | Prod rows: dropped column stripped before return.        |
| `RENAME COLUMN` | Record `old_name → new_name`.         | Hydrated rows: column renamed on insert.           | Prod rows: column renamed in result set.                 |
| `ALTER TYPE`    | Record `column: old_type → new_type`. | Hydrated rows: value cast on insert.               | Prod rows: value cast in result set.                     |
| `CREATE TABLE`  | Mark as Shadow-only table.            | N/A — table doesn't exist in Prod.                 | N/A.                                                     |
| `DROP TABLE`    | Mark as dropped.                      | N/A.                                               | Queries referencing this table return "table not found." |
| `CREATE INDEX`  | Applied to Shadow only. Silent.       | N/A.                                               | N/A.                                                     |

## 9. Transaction Handling

### 9.1 Basic Model

Mori maintains per-connection transaction state.

- `BEGIN`: Open transactions on both Prod (read-only) and Shadow.
- `COMMIT`: Commit Shadow transaction. Close Prod transaction (it was read-only, so this is effectively a rollback/release).
- `ROLLBACK`: Rollback Shadow transaction. Close Prod transaction. Revert any Taint Map / Tombstone additions made during this transaction.

### 9.2 Transaction-Scoped Taint Tracking

Within a transaction, Taint Map and Tombstone additions are staged. On `COMMIT`, they are promoted to the persistent state. On `ROLLBACK`, they are discarded. This ensures that a failed transaction doesn't leave phantom taints.

### 9.3 Isolation

The Prod connection within a transaction should use `REPEATABLE READ` isolation at minimum, ensuring that all reads within a transaction see a consistent snapshot of Prod even if Prod is being mutated concurrently.

## 10. CLI

### 10.1 Commands

```
mori init --from <connection_string> [--engine postgres|mysql|...]
```

Initialize a Mori project. Dumps Prod schema, spins up Shadow, offsets sequences. Creates `.mori/` directory.

```
mori start [--port <port>] [--verbose]
```

Start the Proxy and Shadow. Listens for app connections. `--verbose` logs all intercepted queries and routing decisions.

```
mori stop [--keep-shadow]
```

Graceful shutdown. Persists state. `--keep-shadow` leaves the Shadow container running for direct inspection.

```
mori reset [--hard]
```

Wipe all local state. `--hard` also re-syncs schema from Prod (for when Prod schema has changed).

```
mori status
```

Display current state:

```
Engine:       PostgreSQL 16.2
Prod:         postgres://prod-host:5432/mydb (read-only)
Shadow:       postgres://localhost:54320/mori_shadow
Proxy:        localhost:5432

Tainted Rows:
  users           4 rows
  orders          1 row

Tombstones:
  users           1 row

Schema Diffs:
  users           +phone (TEXT), -fax

Sequence Offsets:
  users.id        start=1,000,001 (prod max: 834,219)
  orders.id       start=1,000,001 (prod max: 612,003)
```

```
mori snapshot save <name>
```

Persist current Shadow data + Taint Map + Tombstone Set + Schema Registry as a named snapshot. Stored in `.mori/snapshots/<name>/`.

```
mori snapshot restore <name>
```

Restore Shadow and all metadata from a named snapshot.

```
mori snapshot list
```

List all saved snapshots with timestamps and dirty-table summaries.

```
mori log [--tail <n>]
```

Stream or dump recent proxy activity — queries, routing decisions, hydrations. Useful for debugging.

```
mori inspect <table>
```

Show detailed state for a table: tainted row count, tombstone count, schema diffs, sequence offset, sample of local rows.

## 11. Open Questions

### Open Question 1: Multi-Table Reads (JOINs) Across Dirty and Clean Tables

**The problem.** A query joins a tainted table with a clean table:

```sql
SELECT o.id, o.total, u.name
FROM orders o
JOIN users u ON o.user_id = u.id
WHERE u.active = true
ORDER BY o.created_at DESC
LIMIT 50
```

If `users` has local mutations but `orders` is clean, neither database can execute this correctly alone. Prod doesn't have the mutated users. Shadow doesn't have any orders.

**Possible approaches:**

**A) Selective Hydration.** Detect the join. Hydrate the necessary rows from the clean table(s) into Shadow, then execute the entire query against Shadow. For the example above: identify the `orders` rows that would match, pull them into Shadow, run the join locally. Correct but potentially expensive — the set of orders that match depends on the join condition and filters, which may require executing a partial query against Prod first to identify candidates.

**B) Decompose and Merge.** Break the join into per-table queries, execute each against the appropriate backend, then perform the join in the Proxy. Works for simple equijoins. Breaks down for complex joins (subqueries, lateral joins, window functions over joined data), and pushes significant query planning into the Proxy.

**C) Escalate to Full Table Sync.** If a query involves a join between tainted and clean tables, bulk-sync the clean table(s) into Shadow and execute entirely locally. Simple logic, but could be slow for large tables and defeats the "zero copy" property.

**D) Restrict and Warn.** Detect the condition, execute against Prod (accepting stale data for tainted rows), and emit a warning to the app log / `mori log`. Pragmatic — most dev workflows tolerate minor staleness. Allows the developer to explicitly hydrate if they need correctness.

**The decision depends on** the target user's tolerance for correctness vs. performance, and how common cross-dirty/clean JOINs are in practice. Most CRUD-heavy apps (the primary use case) tend to query single tables or join tables that are likely to be co-tainted. The adversarial case (complex analytical joins across many tables with mixed taint) may not be worth optimizing for in v1.

---

### Open Question 2: Prod Data Drift During Long-Running Sessions

**The problem.** Prod is live. While a Mori session runs, Prod continues to accept writes from other sources. This creates divergence:

- A row that was read from Prod at time T₀ and hydrated into Shadow may have been updated in Prod at time T₁. Shadow holds the stale T₀ version (plus any local mutations).
- New rows inserted into Prod after session start are visible in non-tainted reads but invisible to Shadow. If one of these new rows has a PK that falls below the sequence offset, it coexists correctly. But relationships to tainted rows may be inconsistent.
- A row deleted in Prod after session start would still appear in Mori reads (Prod query returns nothing, Shadow doesn't have it, row vanishes). This is actually correct behavior — the dev sees a point-in-time-ish view. But it could be confusing.

**Possible approaches:**

**A) Pin to Snapshot.** On `mori start`, begin a long-lived `REPEATABLE READ` or `SERIALIZABLE` transaction (or use Postgres' `pg_export_snapshot()`). All Prod reads see a frozen point-in-time. No drift. Downside: holds a snapshot open on Prod, which can cause issues with vacuuming, replication lag, and connection limits for long sessions.

**B) Accept Drift.** Document that Prod reads are live and may reflect changes made after session start. This is arguably fine — the dev is working against "approximately current prod data," which is already better than a stale local dump. The mental model: Shadow overrides are consistent, Prod reads are best-effort recent.

**C) Configurable Staleness.** Offer both modes: `mori start --snapshot` pins to a point-in-time; `mori start` (default) uses live reads. Let the developer decide based on their workflow.

**The decision depends on** whether Mori's users are primarily running quick ad-hoc tests (where drift is irrelevant) or longer integration test suites (where consistency matters more). The pinned snapshot approach is cleaner semantically but has real operational cost on the Prod database.

## 12. Future Considerations

These are explicitly out of scope for v1 but inform architectural decisions.

- **Multi-engine support.** v1 targets Postgres. The Proxy architecture (protocol adapter → classifier → router) is designed to be engine-pluggable. MySQL, MongoDB, and Redis adapters can be added without changing the core routing/taint/merge logic.
- **Team sharing.** Named snapshots could be pushed/pulled to a shared store, letting teams share pre-configured data states ("here's a snapshot with the edge case that breaks checkout").
- **AI agent integration.** Expose a Mori control API (start/reset/snapshot via HTTP or Unix socket) so that AI coding agents can programmatically manage instances as part of their test-verify loop.
- **Partial schema sync.** `mori init --tables users,orders` to only shadow a subset of tables. Non-shadowed tables are always pass-through to Prod.
- **Read replicas.** Point Mori at a read replica instead of primary Prod to reduce load and risk.
